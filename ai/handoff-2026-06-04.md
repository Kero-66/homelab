## Completed Work

### autobrr — RESOLVED ✅

**Root cause:** The `implementation` column was empty (`""`) for all 6 indexers in the SQLite DB. autobrr source (`indexer/service.go mapIndexer`) defaults `""` to `"irc"`, then `Start()` calls `s.mapIRCServerDefinitionLookup(indexer.IRC.Server, indexer)` where IRC is nil → nil pointer panic on every startup.

**Fix applied:**
```bash
sudo sqlite3 /mnt/Fast/docker/autobrr/config/autobrr.db \
  "UPDATE indexer SET implementation = 'feed' WHERE identifier IN ('torznab', 'torznab-subsplease', 'torznab-shanaproject', 'newznab', 'rss', 'subsplease');"
```

**Additional changes:**
- Pinned image to `ghcr.io/autobrr/autobrr:v1.79.0` (v1.80.0 has same nil-ptr bug, upstream unaware). Committed to repo.
- Compose temporarily uses `ix-arr-stack_default` / `ix-downloaders_default` (actual TrueNAS network names, pending full network migration to plain names)
- autobrr is NOT a TrueNAS midclt-managed app — it is deployed via **Dockhand** (GitOps-style). Deploy via Dockhand API, not midclt.
- autobrr DB owned by `root:root` (644) — needs `sudo sqlite3` to write

**Current state:** autobrr `Up` on v1.79.0, all feeds starting cleanly.

**Fixed so far:**

- Both `imprints.json` cache files touched to appear fresh (bypassing the `mylar3.github.io` download that blocks startup with no timeout)

**Current state:** Still 503. The `_INITIALIZED = True` flag is never set. Startup hangs silently after the versioncheck log entry:
```
WARNING :: MAIN : versioncheck.py:getVersion:347 : No branch specified within config - will attempt to poll version from comicarr
```
No further log entries after this. Quiet logging mode suppresses errors, so the blocking call is invisible.

**Investigation so far:**
- `sab_host = None` → SABnzbd version check won't block
- `check_github = False`, `check_github_on_startup = False` → GitHub version check won't block
- `cache_dir = None` in config.ini → app uses computed default (`/config/comicarr/cache/`)
- Both `imprints.json` files are fresh (touched at 23:41)
- `INITIALIZED` is set at `__init__.py:734`, after which the startup completes
- Last investigation was tracing what `getVersion` calls lead to after it returns — was interrupted
